# GCP 節費檢查 Skill

協助使用者檢查 GCP 帳單、找出可省的開銷。**本 skill 僅允許唯讀操作（`list` / `describe` / `show`），禁止任何修改、刪除、關閉、變更資源的指令。**

## 前置步驟

執行此 skill 前，先用 Skill tool 觸發 `/cli` skill，載入 CLI 相關規則，JSON 輸出用 jq 處理。

## 核心方法論

判斷「能不能省」看的是**當前資源狀態**：

- **Billing 是「帳本」**：只回答「錢花在哪個費用項目（SKU）、花了多少」。看不到有多少資源、資源狀態。
- **`gcloud`（主力）查「即時資產狀態」**：實際列出資源狀態。**所有節費判斷的依據都在這裡。**

所以不要回頭去 billing export 撈更細的用量數字——用量不會告訴你哪個具體資源能砍。直接把費用項目對應到資源類型，用 `gcloud` 唯讀查即時狀態。

**費用大小與趨勢各司其職**：費用大小決定「先查哪個」（大的先看，省的空間大）；趨勢異常代表「有問題」，要優先追異常，不論費用大小。哪個 SKU 在漲直接問使用者（不用為此跑 BigQuery）。但無論哪個訊號指向哪裡，最終「砍哪個資源、怎麼省」的判斷都落在 `gcloud` 的即時資源狀態。

## 執行流程

1. **向使用者釐清輸入**（缺哪項問哪項）：哪個 project、有哪些費用項目各花多少（決定先查哪個）、哪個在漲（優先追的異常）。
2. **確認 project 與存取權**（免費）：
   ```
   gcloud config get-value project
   gcloud projects describe <PROJECT_ID>
   ```
3. **把費用項目對應到服務與資源類型**，用 `gcloud` 唯讀列出該類資源的清單與狀態（`list` / `describe`，皆免費 metadata 讀取）。具體查什麼欄位、判斷點是什麼，當場依該類資源決定，不預先寫死。
4. **從資源狀態找節費目標**，整理成「發現的浪費 / 可換的方案 + 預估節省」呈現。**只提建議、附判斷依據，不執行任何變更**——變更是破壞性操作，交回使用者決定。

## 結論輸出格式（SRE monthly invoice review note）

節費檢查的結論若要貼進 SRE 的月度帳單 review issue，沿用該 issue notes 既有的 per-project 段落格式。

每個 project 一段，結構如下：

```markdown
## <project-id>

- <整體趨勢：費用持平／某段時間顯著上升，附實際觀察到的區間與日期，不要預設月數>
- 費用最高的 service 是 <service1, service2, ...>（由高到低）
  - <某 service 內費用最高的 SKU 列表>
    - <逐項判斷：是否特別／正常；對到哪個實際資源（附 console 或監控圖連結）；能不能省、省多少；結論通常是「先觀察就好」或指出可行動項>

### 待確認 / 後續

- <需要跟人確認、或本次未探底、留待下次補查的項目>
```

寫作慣例：

- **務實保守**：多數項目落在「後續再觀察」；只有確定的浪費或明確異常才提行動，不確定就寫進「待確認」。
- **每個判斷都帶出處連結**：Grafana 監控圖、GCP console 資源頁、相關 Slack thread、其他 issue。呼應「前提要有出處」——費用異常要能連到對應的更新紀錄或討論。
- **既有設計不算浪費**：刻意的設計（如異地 archive snapshot）查到出處後標「維持現狀」，不列可省項。
- **SKU → 資源要對清楚**：說明每個 SKU 對到哪些資源，並附使用率佐證來估節省。跨 project 比對同 service／SKU 量級判斷是否合理。
