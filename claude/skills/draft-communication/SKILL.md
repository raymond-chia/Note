# Draft Communication

## Overview

使用者描述想溝通的事情後，AI 先問清楚、產生草稿、與使用者來回修改，直到使用者滿意為止，再輸出最終版本。

## 流程

### 第一步：用 AskUserQuestion tool 問清楚三件事

收到使用者的需求後，**必須使用 `AskUserQuestion` tool**（互動式問題介面）來確認以下三點（若使用者已提供則略過對應問題）。不可用純文字列出問題請使用者回答。

一次最多問 4 個問題，可以一次送出所有未知的問題：

- **要傳給誰？** 選項：主管、遊戲團隊、開發者、其他（讓使用者輸入）
- **動機是什麼？** 讓使用者自由輸入為什麼現在要說這件事、背景是什麼
- **對方看完後要做什麼？** 選項：知道就好、做決定、提供意見、採取行動、其他

### 第二步：產生草稿

根據答案，先判斷使用者想溝通的事情中，是否含有多種技術選項（例如：用 A 技術 vs. 用 B 技術、自建 vs. 採購第三方方案、分批上線 vs. 一次切換）。

- **若有多種技術選項**：草稿中直接列出各選項及其優缺點，讓收件人閱讀後能做出選擇。不可自行刪減到只剩一種。
- **若只有一種做法**：直接給出草稿，並說明各段的寫法理由（讓使用者可以提出具體修改意見）。

### 第三步：來回修改

根據使用者的回饋修改，反覆迭代直到使用者說「可以」或「送出」。

### 第四步：輸出最終版本

輸出乾淨、可直接複製貼上送出的最終版本，不附任何說明或標記。

## 各場景的內容結構

### 向主管確認 / 提案

```
【背景】為什麼現在提出這件事
【現況 / 我的理解】目前掌握到的資訊，以及我認為的方向
【問題 / 提案】具體要確認的事，或建議的方向
【需要你做什麼】例：確認是否可行、拍板方向
```

### 向分析師 / 遊戲團隊詢問

```
【背景】為什麼問這個（動機必須寫出來，不能省略）
【具體問題】
【限制 / 情境】讓對方理解範圍或取捨
【需要你做什麼】
```

### Issue 留言更新進度

```
【做了什麼】具體行動
【結果】成功 / 失敗 / 待確認
【下一步】我接下來打算怎麼做，或需要誰做什麼
```

### 發現問題 / 回報異常

```
【現象】觀察到什麼
【推測原因】（若有）
【已做的事】
【待確認 / 待處理】
```

## 品質標準

每次草稿都必須符合以下條件，否則主動重寫再給使用者看：

- 動機清楚寫出來，不讓對方猜
- 陳述現狀與提案分開，不混在一起
- 推論有完整的邏輯鏈，不跳躍
- 對方要做什麼，明確說出來
- 若有策略或選項，先說明取捨再讓對方選
